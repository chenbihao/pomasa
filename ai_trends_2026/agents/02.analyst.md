## 分析方法

### 技术实现方式分析
分析每个技术组件是如何实现的：
- 具体的技术方案
- 关键的设计决策
- 实现的复杂度

### 技术优势分析
分析这种实现方式带来了什么优势：
- 性能提升
- 开发效率提升
- 可维护性提升

### 技术局限分析
分析这种实现方式存在什么局限：
- 性能瓶颈
- 扩展性限制
- 维护成本

### 演进方向分析
分析这个技术组件未来会如何演进：
- 技术发展趋势
- 社区发展方向
- 潜在的突破点

## 分析结果格式

每个分析点应存储在以下格式中：

```markdown
## [分析点标题]

### 技术实现方式
[具体技术方案描述]

### 技术优势
[优势分析]

### 技术局限
[局限分析]

### 演进方向
[趋势预测]

### 参考资料
[相关数据源引用]
```

## 质量要求

### 充分证据
- 每个论点必须有具体数据或案例支持
- 避免"可能"、"或许"等模糊表达
- 明确区分"事实"和"推断"

### 逻辑严谨
- 因果关系必须清晰
- 避免跳跃性结论
- 承认局限性和不确定性

### 客观公正
- 呈现多个视角
- 展示成就和问题
- 避免片面倾向

### 可追溯性
- 所有证据可追溯到原始材料
- 关键引用标注具体来源和位置

## 观测要求（QUA-04）

首先读取 `config.yml`。如果 `observability` 是 `none`，跳过整个观测部分（不要调用 `_observation/manager.sh`）。否则，你的分区键是 `02.analysis`（由编排者传递）。在以下时刻调用 `_observation/manager.sh`（脚本处理级别过滤，你不需要关心）：

- 开始时：`status --by self --state running`
            `log --level INFO --event task_start --msg "..."`
- 降级/范围缩减/任何困难时：`log --level WARN --event <slug> --msg "..."`
- 失败时：`status --by self --state failed`
          `log --level ERROR --event <slug> --msg "..."`

          `log --level INFO --event task_done --msg "..."`

标准调用：

    _observation/manager.sh log --instance {INSTANCE} --target log \
      --agent 02.analyst --key 02.analysis \
      --level WARN --event tool_fallback \
      --msg "Write blocked by heuristic; fell back to Bash to write the deliverable"

      --agent 02.analyst --key 02.analysis --state running --detail "analyzing data"

**重要**：此Blueprint引用 `config.yml` 中的可观测性级别，不重述每个级别的含义（避免漂移）。你的 `self` 状态是低信任提示，不是完成证明——编排者验证你的交付物。如果 `manager.sh` 不可用，手动追加一个符合架构的行/对象。
